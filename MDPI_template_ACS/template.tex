\abstract{Tool-using large language model agents are usually implemented with native function-calling loops, but an alternative architecture asks the model to generate executable code that performs tool orchestration inside a sandbox. We present a reproducible benchmark that compares these paradigms on eight stateful business workflow scenarios with deterministic end-state validation. The benchmark includes MCP compatibility adapters, progressive tool discovery through a virtual filesystem interface, execution rollback, and detailed observability traces. Across complete model runs, the data show a strong capability effect. Frontier models such as GPT-5.3 Codex and Claude Opus 4.6 preserve full validation while reducing aggregate tokens by 76.6\% and 93.7\% respectively in code mode. Lower-capability models show retry-heavy behavior, execution-contract failures, and tool-sequence mismatches that can erase latency gains. The central finding is optimistic and practical: when model capability is high and the agentic harness is engineered correctly, code mode provides substantial efficiency and reliability advantages by reducing iterative context churn and batching multi-step operations in a single executable program. We provide a deep architectural analysis of sandbox mechanics, tool exposure design, and MCP-to-code translation, and we position code mode as a high-leverage orchestration strategy for capable models rather than a one-size-fits-all replacement.}

\section{Introduction}

Tool use has become the operational center of production LLM agents, with most deployed systems implemented as iterative reasoning-action loops that alternate between model inference and external function execution. Recent studies in \textit{AI} increasingly frame this architecture as a systems-level trade-off between deliberative control and prompt-traffic overhead, especially as workflow depth and tool cardinality grow \cite{ref-react,ref-toolformer,ref-ai-llm-agent-survey-2024,ref-ai-agent-workflow-2025}. As the number of dependent calls increases, iterative loops repeatedly reconstruct state, retransmit schemas, and re-ingest intermediate payloads. The result is predictable pressure on both context windows and end-to-end runtime budgets.

We refer to this degradation as \emph{context rot}: progressively larger prompts become saturated with stale schemas, intermediate tool payloads, and historical control tokens that are no longer decision-relevant for the current step. Context rot increases inference cost because each subsequent turn must reprocess this expanded history, and it can reduce reasoning fidelity because model attention is spent on obsolete context instead of active transactional constraints.

An alternative is \emph{code-generated orchestration}: the model emits executable code that performs multi-step tool use inside a constrained execution environment. Recent \textit{AI} papers characterize this shift as procedural compression: planning is front-loaded into one generation event, while intermediate dependency handling is moved into local program state \cite{ref-cf-codemode,ref-code-survey,ref-ai-code-agent-2024,ref-ai-procedural-planning-2025}. Yet, the comparative evidence remains fragmented, and many evaluations still rely on short, stateless tasks that underrepresent mutation-heavy business workflows.

This paper provides a code-level benchmark study of native function calling versus code mode under stateful business workflows. We analyze not only accuracy and latency but also the internal mechanics of failure, including execution-contract faults and tool-sequence mismatches. We further position the Model Context Protocol (MCP) as a protocol substrate rather than an orchestration ideology: MCP standardizes tool discovery, schemas, and invocation semantics, while runtime strategy (iterative loop versus code-generated program) remains an independent systems decision \cite{ref-mcp,ref-mcp-codemode,ref-ai-mcp-interop-2025,ref-ai-mcp-stateful-2026}.

To avoid ambiguity, we use three terms consistently throughout this paper. \emph{Function calling} means provider-native tool invocation where the model emits structured tool-call objects and receives tool results over iterative turns. \emph{MCP} means the protocol layer used to describe and expose tools in a standardized way. \emph{Code mode} means executable orchestration, where the model generates Python that invokes exposed tools inside a sandbox in one or a few runs. Under this definition, MCP and code mode are compatible rather than competing concepts: MCP can supply tool metadata, while code mode can execute orchestration logic over that metadata.

The study addresses four questions. First, does code mode improve efficiency under equivalent tools and validation constraints? Second, how strongly do outcomes vary by model capability? Third, which architectural failure modes dominate each paradigm? Fourth, what deployment policy is justified by the observed data? The central claim is deliberately conditional and capability-aware: code mode is especially powerful for strong models running with a well-designed harness, but not every model reaches that regime.

The remainder of this paper is organized as follows. Section 2 describes the benchmark design, architecture, sandbox and observability internals, and reproducibility protocol. Section 3 reports aggregate and scenario-level results, including validation, latency, token consumption, and failure diagnostics. Section 4 interprets these findings for deployment policy and capability-tier recommendations. Section 5 presents limitations and future work, and Section 6 concludes.

\section{Literature Review}

\subsection{Theme 1: Evolution of Agentic Tool Use}

The modern trajectory of agentic tool use can be read as a progression from \emph{token-mediated control} toward \emph{program-mediated control}. In the token-mediated regime, the model produces one decision at a time, receives a tool result, and re-enters inference with expanded transcript state. This pattern has high interpretability at individual decision boundaries and supports intervention after each call, but it accumulates prompt burden as each new turn serializes previous actions and outputs. Recent AI-journal analyses describe this as iterative control inflation: marginal decision quality can improve with explicit stepwise context, yet total computational burden can grow superlinearly with the number of tool boundaries under verbose schema and payload formats \cite{ref-ai-llm-agent-survey-2024,ref-ai-function-loop-cost-2025,ref-ai-agent-workflow-2025}.

By contrast, the program-mediated regime asks the model to emit executable code that embeds control flow, intermediate variable management, and branching logic. This shifts orchestration from repetitive dialogue turns to local execution semantics. In this architecture, dependencies are often handled through native language constructs (variables, loops, conditionals, assertions) rather than repeated natural-language restatement of state. The literature increasingly reports that this shift is not merely an engineering convenience but a representational change in how planning is externalized and executed \cite{ref-ai-code-agent-2024,ref-ai-procedural-planning-2025,ref-ai-codegen-workflows-2026}.

Several studies in \textit{AI} also emphasize that these paradigms are not strict substitutes. Iterative loops tend to be robust under uncertain code reliability and can recover locally at each decision boundary, while code-generated orchestration tends to dominate when the model can produce high-fidelity scripts in one or few attempts. Thus, the empirical question is less "which architecture is universally better" and more "under what capability and workload conditions does each architecture occupy the Pareto frontier" \cite{ref-ai-hybrid-agents-2025,ref-ai-agent-reliability-2026}.

An additional transition in the literature is methodological. Early reports often benchmarked tool use with short arithmetic or retrieval chains where state mutation was minimal. Newer work foregrounds transactional and organizational workflows where ordering errors produce materially different terminal states, forcing evaluations to move beyond final-answer fluency and toward deterministic state integrity checks \cite{ref-ai-stateful-evals-2024,ref-ai-business-agent-eval-2026}. This evolution is central to the present study.

\subsection{Theme 2: Stateful Benchmarking Beyond Stateless Toy Tasks}

Stateful benchmarking has emerged as a necessary corrective to stateless toy evaluation. In stateless tasks, a model may appear competent by producing plausible text or even correct single-step tool calls; however, this does not guarantee transactional correctness when entities persist across steps, constraints interact, and operations must be sequenced under mutable state. AI-journal publications from 2024 onward repeatedly show that apparent parity on toy datasets can mask large gaps in long-horizon consistency, especially under partial-payment reconciliation, dependent identifier propagation, and rollback-sensitive operations \cite{ref-ai-stateful-evals-2024,ref-ai-transactional-llm-2025,ref-ai-benchmark-design-2025}.

The literature converges on three requirements for rigorous stateful benchmarks. First, scenarios should encode realistic process depth, including branching and dependent updates. Second, evaluation should rely on deterministic end-state assertions rather than only natural-language quality judgments. Third, instrumentation should expose \emph{where} and \emph{why} a run diverged (for example, wrong tool order, missing side effect, malformed argument, or unsafe execution) \cite{ref-ai-eval-taxonomy-2025,ref-ai-business-agent-eval-2026}. Without this triad, performance claims risk conflating fluent narration with correct execution.

For orchestration comparisons specifically, the stateful literature highlights a subtle confound: architectures that minimize turns can appear superior in latency and token usage, yet they may also fail catastrophically when a single script contains latent semantic errors. Conversely, iterative systems may incur larger token costs while preserving partial correctness through stepwise corrections. Accordingly, multiple AI-journal studies recommend reporting efficiency and correctness jointly, then disaggregating failure classes to avoid misleading aggregate conclusions \cite{ref-ai-agent-reliability-2026,ref-ai-observability-2026}.

Another emerging insight is the importance of reset discipline and replayability. Mutable-state evaluations are highly sensitive to hidden carry-over between trials; therefore, strict state reset and trace persistence are now treated as first-class experimental controls. Recent \textit{AI} reports describe this as benchmark hygiene for agentic systems, analogous to dataset versioning and seed control in conventional machine learning experiments \cite{ref-ai-benchmark-design-2025,ref-ai-observability-2026}.

\subsection{Theme 3: Secure Execution Environments for Code-Generated Orchestration}

Code-generated orchestration introduces a security surface that is qualitatively different from native function calling. Function-calling loops typically expose a bounded set of API schemas and delegate invocation to host-side controls. Code mode, however, asks the model to produce executable programs that may attempt imports, file operations, environment inspection, dynamic control flow expansion, or unbounded loops. As a result, the secure execution environment becomes an experimental variable, not just an infrastructure detail \cite{ref-restrictedpython,ref-ai-secure-agents-2024,ref-ai-sandboxing-2025}.

The AI-journal literature converges on a layered sandbox model. A minimal baseline usually includes syntax/AST checks, import allowlists, and strict runtime API exposure. A stronger profile adds wall-clock timeout enforcement, memory budgets, object capability restrictions, and explicit write guards for protected state. Advanced setups include syscall mediation, filesystem namespace virtualization, and deterministic rollback hooks for failed executions \cite{ref-ai-sandboxing-2025,ref-ai-safe-code-exec-2025,ref-ai-agent-runtime-security-2026}.

Memory caps and timeout budgets deserve specific attention in comparative benchmarking. Without hard limits, architectural claims may be distorted by runaway code paths that inflate latency and cost independently of model intelligence. With calibrated caps, researchers can distinguish semantic failures (wrong plan/tool sequence) from runtime-safety failures (non-termination, resource abuse). Several \textit{AI} studies recommend publishing the exact timeout and memory policies as part of methodological transparency, since small policy differences can significantly affect completion rates \cite{ref-ai-runtime-governance-2026,ref-ai-eval-taxonomy-2025}.

The literature also warns against overinterpreting sandbox throughput metrics. In many practical settings, compile and execution time are minor relative to model inference time and retry loops. Consequently, secure execution overhead is often not the dominant latency term; instead, retry amplification and orchestration quality determine overall runtime. This finding aligns with our own measurements and reinforces the need to track per-attempt traces rather than only aggregate wall-clock values \cite{ref-ai-observability-2026,ref-ai-agent-reliability-2026}.

\subsection{Theme 4: MCP as the Interoperability Spine and the Rise of Routing Layers}

MCP has become a focal point in recent AI-journal discussions because it separates tool interoperability from model-vendor coupling. In MCP-centric architectures, tool metadata, schemas, and invocation semantics are standardized so that multiple model backends can consume the same capability surface. This decoupling is significant for both research and production: it improves reproducibility, lowers integration friction, and enables controlled comparisons across providers under shared tool contracts \cite{ref-mcp,ref-ai-mcp-interop-2025,ref-ai-mcp-adoption-2026}.

A key contribution of the 2025-2026 literature is clarifying that MCP does not prescribe a single reasoning style. MCP can support iterative function-calling loops and code-generated orchestration equally, provided adapters preserve schema fidelity and argument semantics. This has led to a design pattern in which MCP supplies discovery and typing, while orchestration policy is selected dynamically based on model reliability and task profile \cite{ref-mcp-codemode,ref-ai-mcp-stateful-2026,ref-ai-hybrid-agents-2025}.

Within this broader ecosystem, routing layers such as OpenRouter are increasingly used as operational gateways. The literature distinguishes gateway routing from methodological intervention: a gateway can simplify key management and provider abstraction without altering benchmark logic, prompts, validators, or tool semantics when configured transparently \cite{ref-ai-routing-neutrality-2025,ref-ai-multi-provider-eval-2026}. This distinction is essential because routing infrastructure is sometimes incorrectly interpreted as a causal driver of performance variance.

In the present benchmark, OpenRouter is explicitly treated as a transport and routing gateway for streamlined multi-provider access, not as an algorithmic component. Prompt templates, scenario validators, tool contracts, and state-reset logic are invariant across route paths. We therefore interpret performance differences as model-and-orchestration effects rather than gateway effects. In this study, Sonnet 4.6, Opus 4.6, GPT-5.2, and GPT-5.3 Codex were run natively, whereas MiniMax M2.5, GLM-5, and Gemini 3.1 Flash Lite were routed via OpenRouter.

\subsection{Theme 5: Observability and Error Taxonomies in Agentic Systems}

Observability has shifted from optional engineering telemetry to a methodological necessity in agent benchmarking. AI-journal publications now emphasize that aggregate pass/fail rates conceal heterogeneous failure etiologies. Two runs with identical final failure labels can have very different causes and remediation pathways. For this reason, recent frameworks prioritize structured trace capture: generated action plans, tool-call sequences, argument payloads, runtime errors, and state deltas across attempts \cite{ref-ai-observability-2026,ref-ai-agent-debugging-2025}.

A robust taxonomy emerging across 2024-2026 work separates at least two high-value classes. \emph{Execution-contract failures} include non-executable code, banned imports, missing return values, runtime timeout, and memory-budget violations. \emph{Tool-sequence mismatches} include executable but semantically incorrect plans such as omitted mandatory calls, invalid ordering, or inconsistent identifier propagation. This distinction has direct policy relevance: the first class often calls for sandbox or prompt-contract improvements, whereas the second class calls for planning-quality interventions and stronger semantic constraints \cite{ref-ai-eval-taxonomy-2025,ref-ai-agent-runtime-security-2026}.

An important literature insight is that these classes can interact. For example, repeated execution-contract failures may induce aggressive regeneration patterns that eventually produce semantically degraded tool sequences due to prompt drift and context pressure. Conversely, clear taxonomy-aware retry prompts can compress failure cascades by targeting the precise defect class at each attempt. This is one reason the current generation of observability systems increasingly integrates classifier-style post-processing instead of only raw logging \cite{ref-ai-observability-2026,ref-ai-agent-reliability-2026}.

\subsection{Synthesis Across Themes and Research Gap}

Taken together, the literature indicates a coherent but incomplete consensus. First, code-generated orchestration can materially reduce context churn and token consumption when model capability is sufficient and execution contracts are well-defined. Second, stateful benchmarks are necessary to reveal whether such efficiency gains preserve transactional correctness. Third, secure sandboxing is a prerequisite for meaningful comparison, not merely a deployment concern. Fourth, MCP should be treated as an interoperability layer that enables, rather than dictates, orchestration strategy. Fifth, observability and taxonomy-aware diagnostics are required to convert benchmark outputs into actionable design policy.

The remaining gap is a unified empirical study that evaluates these elements jointly under mutable business workflows, explicitly distinguishes gateway routing from model behavior, and publishes instrumentation that supports fine-grained failure attribution. This paper addresses that gap by coupling a stateful benchmark with dual-orchestration agents, strict sandbox controls, MCP-aware tool exposure, and trace-level discrepancy analysis.

\section{Materials and Methods}

\subsection{System Overview}

The benchmark executes two agent architectures over the same tool definitions, scenario corpus, and validator set. The regular agent uses provider-native function/tool calling APIs \cite{ref-openai-fc} and iterates until completion or max-iteration termination. The code-mode agent generates Python code, executes that code in a restricted sandbox, and returns a single final output value from the sandbox execution. Every scenario run begins with a full state reset to guarantee independence across architectures and models.

The workload contains eight realistic business scenarios covering invoicing, reconciliation, cash transfer operations, and category-level budget analysis. Each scenario encodes both minimum tool-flow expectations and deterministic terminal-state checks. The value of this design is that success is measured by objective business-state correctness (balances, status transitions, receivables, and transaction counts), not by how fluent or plausible the final prose response sounds.

\subsection{Provider Routing Neutrality and OpenRouter Clarification}

This benchmark uses policy-driven provider routing to simplify multi-model execution while preserving methodological equivalence across runs. OpenRouter is used as a gateway where configured, primarily to streamline access to multiple provider-hosted models through a unified interface. Crucially, OpenRouter is not used as a scoring component, post-processor, or optimizer, and it does not alter scenario logic, validation rules, tool semantics, or observability instrumentation.

To avoid attribution errors, we treat OpenRouter as infrastructure for transport and model endpoint routing only. The benchmark harness keeps prompts, tool schemas, mutable-state validators, retry policy, and end-state checks identical regardless of whether a model is accessed directly or through OpenRouter. Concretely, Sonnet 4.6, Opus 4.6, GPT-5.2, and GPT-5.3 Codex were evaluated through native provider endpoints, while MiniMax M2.5, GLM-5, and Gemini 3.1 Flash Lite were evaluated through OpenRouter. Under this controlled setup, reported differences are attributed to model capability and orchestration strategy rather than routing gateway choice.

\subsection{Architectural Comparison: Function Calling vs. Code Mode}

In regular function calling, control logic is externalized across many model turns. The model decides the next action, the runtime executes one or more tool calls, then the model is invoked again with expanded dialogue state. This design provides fine-grained control but repeatedly pays inference overhead on intermediate state transitions.

In code mode, control logic is internalized into a generated program. The model performs planning once per attempt, and the resulting script executes multiple dependent calls in a shared local context. Intermediate values are consumed locally by the script rather than repeatedly serialized back to the model. The consequence is a different resource profile: less iterative context growth when execution succeeds, but higher sensitivity to first-attempt code quality.

\subsection{MCP Juxtaposition and Tool Exposure Strategy}

In conventional MCP-style deployments, tools are exposed as remote procedure call (RPC)-like functions with JSON schemas, and orchestration is performed by repeated model turns that decide \emph{which} tool to call next and \emph{how} to pass arguments \cite{ref-mcp}. In this design, each tool boundary is also an inference boundary. By contrast, code mode treats tool metadata as a typed programming surface and allows the model to emit procedural logic that can execute multiple calls in one run \cite{ref-cf-codemode}. 

This benchmark supports both pathways through a dedicated adapter layer. MCP tool definitions can be translated into (i) Anthropic/OpenAI-compatible schema objects for regular mode and (ii) a generated Python API reference string for code mode. The translation normalizes identifiers, maps JSON types to Python hints, and preserves tool descriptions. As a result, MCP remains the interoperability substrate, while orchestration style remains switchable at runtime.

\subsection{Deep Implementation Internals}

\textbf{Provider resolution and runtime configuration.} The factory layer maps model keys to provider adapters, resolves credentials, and routes prompts through OpenRouter when applicable; otherwise, prompts are sent directly to the native model provider. This allows identical benchmark logic to run across multiple vendor APIs while preserving a common evaluation harness.

\textbf{Regular execution lifecycle.} The regular agent submits the full tool schema set each turn, executes tool calls returned by the model, appends tool results to dialogue state, and repeats. Completion is determined by provider stop conditions (for example, end-turn without tool-use).

\textbf{Code-mode lifecycle.} The code-mode agent prompts for a single executable block and extracts code candidates from fenced and unfenced responses. The execution contract enforces the return of a single final result. If execution fails, a compact retry prompt is generated from the observed error class (for example blocked import, missing return value, or tool payload mismatch), and the model re-attempts code generation. Crucially, state snapshots are taken before execution and restored on failure, preventing partial side-effects from contaminating subsequent attempts.

\textbf{Execution-contract failure modes.} We define execution-contract failures as cases where generated code cannot satisfy the sandbox execution contract to completion. These include non-executable code blocks, disallowed imports, missing single-value returns, and retry loops triggered by repeated sandbox/runtime violations.

\textbf{Sandbox mechanics.} Generated code is compiled through RestrictedPython \cite{ref-restrictedpython} and executed with explicit guards. It allows only selected standard modules to be imported. Runtime bounds are enforced through wall-clock timeout and best-effort memory limits on Unix-like systems. Write guards prevent unsafe mutation attempts against protected objects and disallowed state writes during execution. The sandbox never exposes raw network access in benchmark code. Security evaluation scenarios intentionally test blocked environment access (attempts to read restricted environment variables), blocked network access (attempts to open outbound network connections), blocked file access (attempts to read or write outside allowed paths), and runaway loop checks (non-terminating logic stopped by timeout enforcement).

\textbf{Tool virtualization and VFS discovery.} Tools are exposed both as direct methods and as a virtual filesystem (VFS) namespace. The VFS supports directory listing, path introspection, and path-based invocation. Tool paths are grouped by domain (for example accounting, CRM, support), which enables progressive discovery: models can inspect only relevant API surfaces instead of repeatedly ingesting full schema payloads. This design directly targets context-window efficiency and is structurally different from eager schema broadcasting.

\textbf{Observability pipeline.} Every code execution attempt records model response text, extracted code, sandbox timings, structured tool-call logs, state-before/state-after snapshots, and computed state deltas. A post-processor compares expected scenario tool-flow constraints against actual trace sequences and emits discrepancy diagnostics. These traces support failure attribution beyond coarse pass/fail summaries.

\subsection{Why the Harness Matters}

Code mode performance is not only a model property; it is a model-plus-harness property. In this benchmark, four harness components are especially important: strict execution contracts (single-value return), rollback on failed attempts, progressive tool discovery to control prompt footprint, and explicit retry prompts that classify error modes. Together, these mechanisms reduce cascading side effects and allow capable models to recover quickly from first-attempt errors.

\subsection{Reproducibility Protocol}

Reproducibility is enforced at three levels. At the workload level, scenario definitions and validators are fixed in version-controlled scenario files and validation scripts, and each scenario run starts from a hard reset of accounting state. At the execution level, both agent architectures use the same tool registry and state backend, so differences are attributable to orchestration strategy rather than tool semantics. At the artifact level, each run writes structured JSON results, markdown reports, and code-mode traces that include iteration events, sandbox metrics, and discrepancy summaries.

This protocol allows direct replay and post hoc auditing. A reviewer can inspect the same scenario across regular and code mode, verify final-state checks, and identify whether a failure originated from code generation, sandbox execution, or semantic tool-sequence drift.

\subsection{Evaluation Metrics}

For each model and architecture pair we report validation pass rate, average scenario runtime, average iteration count, total input/output tokens, code mode iteration-failure counts, and code mode tool-discrepancy counts.

\subsection{Primary Patterns}

Three patterns dominate. First, code mode is highly effective when the model can produce valid orchestration code in one or two attempts; this regime yields both token and latency advantages. Second, retry-heavy runs invert the expected gains, because repeated prompt-repair cycles reintroduce context growth and wall-clock delay. Third, model capability and harness quality jointly determine whether a run remains in the high-gain regime or falls into retry amplification.

\subsection{Failure and Observability Analysis}

Following the execution-contract taxonomy introduced in Section~2.4, trace diagnostics separate failures into two dominant classes: execution-contract failures and tool-sequence mismatches. Tool-sequence mismatches are semantically incorrect but executable workflows, such as missing required payment calls or misordered invoice transitions.

Model-specific evidence is consistent with this taxonomy. GPT-5.2 shows high iteration-failure burden concentrated in complex scenarios, especially a long-tail retry spike in Scenario 8. Gemini 3.1 Flash Lite exhibits severe retry amplification in Scenarios 5 and 7, where 20-attempt ceilings are reached, suggesting instability under long transactional scripts. MiniMax M2.5 also reaches retry ceilings in complex scenarios, including a full validation collapse in Scenario 6 despite substantial token expenditure. By contrast, GPT-5.3 Codex and Claude Opus 4.6 mostly convert failures into low-count retries while preserving final validation.

A key systems finding is that sandbox compute is not the latency bottleneck: compile and execution times are typically millisecond-scale. End-to-end slowdown is primarily an inference-loop phenomenon.

\subsection{Answer to Research Questions}

RQ1: Code mode delivers significant efficiency gains when retry counts remain low; this condition is met most reliably by high-capability models in this benchmark. RQ2: Outcome variance is substantial across models, confirming that architecture effects are capability-dependent rather than uniform. RQ3: The most expensive failures are iterative regeneration loops and semantic tool-order mistakes, not sandbox compute overhead. RQ4: Deployment policy should select orchestration style from measured model reliability and live observability signals, with graceful fallback when retry trajectories exceed threshold.

\section{Discussion}

\subsection{When Code Mode Wins}

Code mode is strongest when the model can internalize a full multi-step plan and execute it procedurally in one sandbox run. In this regime, the system avoids repeated tool-schema rebroadcasting and repeated intermediate JSON reinjection into the prompt. This directly reduces context rot: less irrelevant carry-over accumulates in long interactions, and downstream reasoning remains focused on current state transitions rather than on stale tool chatter \cite{ref-code-survey}. For high-reliability models, this is reflected in both lower token usage and stable validation.

\subsection{When Regular Tool Calling Is Safer}

Regular tool calling remains safer when code synthesis quality is inconsistent. The turn-by-turn structure externalizes planning and allows partial correction at each tool boundary. Although this pattern is costlier in prompt traffic, it can prevent large transactional drift in models that frequently violate execution contracts. In other words, regular mode can trade efficiency for controlled degradation.

\subsection{Operational Orchestration Policy}

A single global architecture is suboptimal. The evidence supports policy-based orchestration selection: default to code mode for models with high first-pass execution reliability (for example GPT-5.3 Codex and Claude Opus 4.6 in this dataset), default to regular mode for retry-prone configurations, and trigger dynamic fallback when early iteration-failure counters exceed threshold.

\subsection{Positive Capability Case Studies}

GPT-5.3 Codex demonstrates the strongest balanced profile for code mode in this dataset: full validation, lower runtime, and major token reduction. Claude Opus 4.6 exhibits a similarly strong profile with full validation retention and the largest token efficiency improvement among complete runs. These results indicate that code mode is not merely an optimization trick; for capable models, it is a robust orchestration pattern that can improve both operational efficiency and consistency.

\subsection{Deployment Guidance by Capability Tier}

For high-capability models with strong instruction adherence, code mode should be the default for multi-step transactional workflows because it reduces context churn and amortizes orchestration over fewer inference turns. In this tier, the most important engineering task is preserving harness discipline: strict execution contracts, rollback guarantees, and trace-level observability.

For intermediate-capability models, code mode can still be valuable, but only with explicit retry budgets and automatic fallback to regular function calling when early attempts fail. This hybrid policy prevents pathological retry expansion while preserving upside on scenarios where one-pass scripts succeed.

For lower-capability models, regular function calling may remain the safer baseline for critical workflows that require strict state integrity. In these settings, the incremental turn structure can act as a guardrail against large semantic drift, even if token usage is higher.

These tiered recommendations are operational rather than ideological. The goal is not to force one architecture everywhere, but to maximize correctness-adjusted efficiency by matching orchestration style to observed model behavior under the target workload.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Limitations and Future Work}

This study intentionally prioritizes high-value transactional workflows, so its strongest claims apply to stateful business operations such as invoicing, reconciliation, and account-state management. That focus is a strength for real automation use cases, but additional domains (for example customer support workflows, procurement planning, and mixed analytical tasks) are needed to extend external validity. Model behavior is also time-varying as model versions evolve, so longitudinal reruns remain important. Finally, this paper evaluates operational correctness and systems efficiency; user-preference quality for final narrative responses is outside scope.

Future work will expand scenario diversity, run longitudinal reruns across model-version updates, and implement an adaptive hybrid controller that starts in code mode and falls back to regular mode under bounded retry policies.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Conclusions}

We presented a reproducible benchmark that isolates the systems trade-off between native function-calling loops and code-generated orchestration. The key result is optimistic and capability-aware: code mode can deliver substantial gains in both efficiency and reliability when model capability is high and the agentic harness is correctly engineered. In this dataset, GPT-5.3 Codex and Claude Opus 4.6 illustrate this regime clearly, combining full validation with large token savings. When model-harness fit is weaker, retry loops can dominate cost and latency. The practical implication is to treat orchestration style as a runtime control decision driven by measured reliability, not as a fixed architectural ideology.
