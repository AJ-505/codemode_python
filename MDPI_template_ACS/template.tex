\abstract{Tool-using large language model agents are usually implemented with native function-calling loops, but an alternative architecture asks the model to generate executable code that performs tool orchestration inside a sandbox. We present a reproducible benchmark that compares these paradigms on eight stateful business workflow scenarios with deterministic end-state validation. The benchmark includes provider-aware model routing, Model Context Protocol compatibility adapters, progressive tool discovery through a virtual filesystem interface, execution rollback, and detailed observability traces. Across six complete model runs, the data show strong model-conditional effects. GPT-5.3 Codex and Claude Opus 4.6 preserve full validation while reducing aggregate tokens by 76.6\% and 93.7\%, respectively, under code mode. In contrast, routed or lower-reliability configurations exhibit retry-heavy behavior, execution-contract failures, and tool-sequence mismatches that can erase latency gains or reduce validation performance. We provide a deep architectural analysis of sandbox mechanics, tool exposure design, and MCP-to-code translation, and we show that code mode is best interpreted as a conditional systems optimization rather than a universal replacement for function calling. The resulting decision framework supports deployment-time routing between regular and code-based orchestration based on model reliability, provider stability, and live failure telemetry.}

\section{Introduction}

Tool-use has become the operational center of modern LLM agents, with most production systems implemented as iterative reasoning-action loops that interleave model inference and remote tool calls. This pattern is rooted in prior work on reasoning-and-acting trajectories and self-supervised tool invocation \cite{ref-react,ref-toolformer}. The growing evidence that code-pretrained models can better maintain structured intermediate representations further motivates executable orchestration designs \cite{ref-code-commonsense}. However, iterative tool loops incur repeated prompt reconstruction, repeated network round-trips, and repeated re-ingestion of intermediate payloads. As tool count and workflow depth increase, this architecture can create substantial context pressure and latency accumulation.

An alternative is \emph{code-generated orchestration}: the model emits executable code that performs multi-step tool use inside a constrained execution environment. Recent industry reports argue that this approach can improve tool reliability and reduce context overhead by shifting orchestration from token-by-token planning to compiled procedural execution \cite{ref-cf-codemode,ref-code-survey}. Yet, public comparisons are often anecdotal, model-specific, or weakly instrumented.

This paper provides a code-level benchmark study of native function calling versus code mode under stateful business workflows. We analyze not only accuracy and latency but also the internal mechanics of failure, including execution-contract faults, tool-sequence mismatches, and provider-path instability. We also formalize the relationship between Model Context Protocol (MCP) tool exposure and code-based orchestration: MCP is treated as a compatibility and discovery substrate, while execution strategy is treated as an orthogonal systems decision \cite{ref-mcp}.

The study addresses four questions. First, does code mode improve efficiency under equivalent tools and validation constraints? Second, how strongly do outcomes vary by model and routing path? Third, which architectural failure modes dominate each paradigm? Fourth, what deployment policy is justified by the observed data? The central claim is deliberately conditional: code mode can be highly beneficial for some model-provider combinations, but it is not a universal replacement for iterative function calling.

\section{Materials and Methods}

\subsection{System Overview}

The benchmark executes two agent architectures over the same tool backend, scenario corpus, and validator set. The regular agent uses provider-native function/tool calling APIs \cite{ref-openai-fc} and iterates until completion or max-iteration termination. The code-mode agent generates Python code, executes that code in a restricted sandbox, and returns the value assigned to \texttt{result}. Every scenario run begins with a full state reset to guarantee independence across architectures and models.

The workload contains eight realistic business scenarios covering invoicing, reconciliation, cash transfer operations, and category-level budget analysis. Each scenario defines expected minimum tool-flow patterns and deterministic postconditions over terminal state. Validation is therefore state-grounded rather than stylistic.

\subsection{MCP Juxtaposition and Tool Exposure Strategy}

In conventional MCP-style deployments, tools are exposed as RPC-like functions with JSON schemas, and orchestration is performed by repeated model turns that decide \emph{which} tool to call next and \emph{how} to pass arguments \cite{ref-mcp}. In this design, each tool boundary is also an inference boundary. By contrast, code mode treats tool metadata as a typed programming surface and allows the model to emit procedural logic that can execute multiple calls in one run \cite{ref-cf-codemode}. 

This benchmark supports both pathways through a dedicated adapter layer. MCP tool definitions can be translated into (i) Anthropic/OpenAI-compatible schema objects for regular mode and (ii) a generated Python API reference string for code mode. The translation normalizes identifiers, maps JSON types to Python hints, and preserves tool descriptions. As a result, MCP remains the interoperability substrate, while orchestration style remains switchable at runtime.

\subsection{Deep Implementation Internals}

\textbf{Routing and provider resolution.} The factory layer maps model keys to provider adapters and supports direct-provider override where dedicated credentials exist. For selected models, the runtime can switch between routed OpenAI-compatible endpoints and direct vendor endpoints, enabling analysis of routing-path effects independent of prompt design.

\textbf{Regular execution lifecycle.} The regular agent submits the full tool schema set each turn, executes tool calls returned by the model, appends tool results to dialogue state, and repeats. Completion is determined by provider stop conditions (for example, end-turn without tool-use).

\textbf{Code-mode lifecycle.} The code-mode agent prompts for a single executable block and extracts code candidates from fenced and unfenced responses. The execution contract enforces a final \texttt{result} variable. If execution fails, a compact retry prompt is generated from the observed error class (for example blocked import, missing result, or tool payload mismatch), and the model re-attempts code generation. Crucially, state snapshots are taken before execution and restored on failure, preventing partial side-effects from contaminating subsequent attempts.

\textbf{Sandbox mechanics.} Generated code is compiled through RestrictedPython \cite{ref-restrictedpython} and executed with explicit guards. The import path is restricted by root-module allow-listing; only selected standard modules are admitted. Runtime bounds are enforced via wall-clock timeout (\texttt{SIGALRM}/\texttt{setitimer}) and best-effort memory constraints (\texttt{RLIMIT\_AS} soft-limit adjustment on Unix-like systems). Safe builtins are explicitly curated, and write guards are applied through RestrictedPython facilities. The sandbox never exposes raw network access in benchmark code, and security evaluation scenarios include blocked-environment, blocked-network, blocked-file, and runaway-loop checks.

\textbf{Tool virtualization and VFS discovery.} Tools are exposed both as direct methods and as a virtual filesystem (VFS) namespace. The VFS supports directory listing, path introspection, and path-based invocation. Tool paths are grouped by domain (for example accounting, CRM, support), which enables progressive discovery: models can inspect only relevant API surfaces instead of repeatedly ingesting full schema payloads. This design directly targets context-window efficiency and is structurally different from eager schema broadcasting.

\textbf{Observability pipeline.} Every code execution attempt records model response text, extracted code, sandbox timings, structured tool-call logs, state-before/state-after snapshots, and computed state deltas. A post-processor compares expected scenario tool-flow constraints against actual trace sequences and emits discrepancy diagnostics. These traces support failure attribution beyond coarse pass/fail summaries.

\subsection{Evaluation Metrics}

For each model and architecture pair we report validation pass rate, average scenario runtime, average iteration count, total input/output tokens, code-mode iteration-failure counts, and code-mode tool-discrepancy counts. The token-to-cost conversion follows standard linear pricing:

\begin{linenomath}
\begin{equation}
\mathrm{cost}=\frac{T_{in}}{10^6}R_{in}+\frac{T_{out}}{10^6}R_{out},
\end{equation}
\end{linenomath}
where $T_{in}$ and $T_{out}$ are input/output token totals and $R_{in}$, $R_{out}$ are provider rates.

\subsection{Primary Patterns}

Three patterns dominate. First, code mode is highly effective when the model can produce valid orchestration code in one or two attempts; this regime yields both token and latency advantages. Second, retry-heavy runs invert the expected gains, because repeated prompt-repair cycles reintroduce context growth and wall-clock delay. Third, routing path matters: in this dataset, native OpenAI/Anthropic paths are generally more stable under code mode than several routed OpenAI-compatible paths.

\subsection{Failure and Observability Analysis}

Trace diagnostics separate failures into two dominant classes. \emph{Execution-contract failures} include non-executable code blocks, blocked imports, missing \texttt{result}, and repeated sandbox-triggered retries. \emph{Tool-sequence mismatches} include semantically incorrect but executable workflows, such as missing required payment calls or misordered invoice transitions.

Model-specific evidence is consistent with this taxonomy. GPT-5.2 shows high iteration-failure burden concentrated in complex scenarios, especially a long-tail retry spike in Scenario 8. Gemini 3.1 Flash Lite (routed) exhibits severe retry amplification in Scenarios 5 and 7, where 20-attempt ceilings are reached, suggesting instability under long transactional scripts. MiniMax M2.5 also reaches retry ceilings in complex scenarios, including a full validation collapse in Scenario 6 despite substantial token expenditure. By contrast, GPT-5.3 Codex and Opus 4.6 mostly convert failures into low-count retries while preserving final validation.

A key systems finding is that sandbox compute is not the latency bottleneck: compile and execution times are typically millisecond-scale. End-to-end slowdown is primarily an inference-loop phenomenon.

\subsection{Answer to Research Questions}

RQ1 is answered conditionally: code mode is efficient when retry counts remain low. RQ2 shows large cross-model and cross-provider variance, so architecture effects are not portable. RQ3 shows that the most expensive failures are iterative regeneration loops, not sandbox runtime overhead. RQ4 supports adaptive routing: orchestration strategy should be selected dynamically from model reliability priors and live telemetry.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Discussion}

\subsection{When Code Mode Wins}

Code mode is strongest when the model can internalize a full multi-step plan and execute it procedurally in one sandbox run. In this regime, the system avoids repeated tool-schema rebroadcasting and repeated intermediate JSON reinjection into the prompt. This directly reduces context rot: less irrelevant carry-over accumulates in long interactions, and downstream reasoning remains focused on current state transitions rather than on stale tool chatter \cite{ref-code-survey}. For high-reliability models, this is reflected in both lower token usage and stable validation.

\subsection{When Regular Tool Calling Is Safer}

Regular tool calling remains safer when code synthesis quality is inconsistent. The turn-by-turn structure externalizes planning and allows partial correction at each tool boundary. Although this pattern is costlier in prompt traffic, it can prevent large transactional drift in models that frequently violate execution contracts. In other words, regular mode can trade efficiency for controlled degradation.

\subsection{Operational Routing Implication}

A single global architecture is suboptimal. The evidence supports policy-based orchestration selection: default to code mode for models with high first-pass execution reliability (for example GPT-5.3 Codex and Opus 4.6 in this dataset), default to regular mode for retry-prone routed configurations, and trigger dynamic fallback when early iteration-failure counters exceed threshold.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Limitations and Future Work}

The study is bounded by domain and time. Scenarios are concentrated in business/accounting workflows; additional domains are needed before broad generalization. Model and provider behavior are non-stationary, so repeated runs across dates are required for temporal robustness. Cost estimates are token-based and do not include all provider-side queueing effects. Finally, we evaluate operational correctness rather than human preference quality of final prose.

Future work will include repeated-seed evaluations with uncertainty intervals, explicit direct-versus-routed ablations per model family, and an adaptive hybrid controller that starts in code mode and falls back to regular mode under bounded retry policies.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Conclusions}

We presented a reproducible benchmark that isolates the systems trade-off between native function-calling loops and code-generated orchestration. The core outcome is not a universal winner but a conditional deployment rule. Code mode can provide very large efficiency gains for models that reliably satisfy executable orchestration contracts. When reliability degrades, retry loops dominate cost and latency, and validation may regress. The practical implication is clear: orchestration style should be a runtime routing decision informed by model-specific reliability and live observability.
